# Tech_Storage — 작업 지침

이 저장소는 기술 소개/기술동향 문서를 축적해 GitHub(public)에 공개하고 슬랙에 공유하는 용도. 새 세션에서도 아래 규칙을 그대로 따를 것.

## 문서 작성 절차 (매번 이 순서로)

1. **주제 확정** — 사용자가 URL/이름만 던지면, 그게 뭔지부터 조사(Agent/WebSearch)해서 확인 후 진행. 지레짐작 금지.
2. **심층 리서치** — Agent(subagent)로 조사. 링크·통계·정량 수치는 반드시 **별도 검증 서브에이전트**로 fact-check 후 인용. URL을 지어내지 말 것.
3. **문서 구조**: 왜 필요한가(문제 상황) → 비유로 이해 → 개념/핵심 설명 → (있으면) 실전 사례 → 다른 도구/사례 비교 → 핵심 원칙 → 참고자료. 정의부터 시작하지 말고 동기(왜 필요한가)부터.
4. **다이어그램**: mermaid로 핵심 흐름 1~3개 포함. 각 다이어그램 아래에 "**그림 읽는 법:**" 캡션으로 그림을 말로 풀어줄 것.
5. **비전문가 접근성 + 깊이를 동시에** — 하나를 위해 다른 하나를 희생하지 않는다. 전문용어는 처음 나올 때 괄호로 풀이.
6. **불확실한 내용은 명시적으로 구분** — 사실처럼 포장하지 말 것. `⚠️ 여기부터는 검증된 사실이 아니라 추정입니다` 같은 경고 블록으로 분리.
7. **특정 도구 중심 서술 피하기** — 가능하면 일반 개념을 먼저 설명하고, 특정 제품/도구는 "구현 사례 중 하나"로만 인용.
8. **문서 맨 끝에 항상 추가**:
   ```
   ---
   📎 더 많은 기술동향: https://github.com/21-Arbiter/Tech_Storage
   ```

## 폴더 구조

```
기술동향/
  <주제폴더>/          # AI-에이전트, LLM-모델, 개발도구, 클라우드-인프라, 보안 (필요시 추가)
    README.md          # 그 폴더의 범위 설명 + 문서 목록
    YYYY-MM-DD-주제.md
```

새 문서 추가 시 **해당 폴더 README.md 목록**과 **루트 README.md 목차**에 둘 다 링크 추가할 것 (둘 중 하나만 하고 끝내지 말 것 — 자주 빠뜨리는 실수).

## 슬랙 공유

슬랙은 GitHub 마크다운이 아니라 mrkdwn을 씀(`**굵게**`→`*굵게*`, 표/mermaid 렌더링 불가).

**현재 실제로 쓰는 방식** (전문 변환보다 이쪽을 기본으로): 전체 문서를 변환해서 붙여넣지 않고, *짧은 개조식 요약(핵심 요약 + 이런 분들께 도움됨) + GitHub 링크로 유도*하는 티저 메시지를 슬랙 mrkdwn으로 직접 작성한다. 다이어그램·표가 있는 상세 내용은 "아래에서 확인하세요" 하고 링크로 넘긴다.

**링크 작성 규칙 (중요, 실수하기 쉬움):**
- 파일 경로에 한글이 들어가면(예: `기술동향/AI-에이전트/...md`) 반드시 URL 인코딩할 것. 인코딩 안 하면 404 남.
  ```bash
  python3 -c "from urllib.parse import quote; print(quote('기술동향/AI-에이전트/2026-07-21-파일명.md'))"
  ```
  결과를 `https://github.com/21-Arbiter/Tech_Storage/blob/main/<인코딩된 경로>` 뒤에 붙인다.
- 슬랙 링크는 `<url|텍스트>` 형식을 쓰지 말고 **URL만 그대로 노출**한다 (텍스트를 붙이면 파싱 오류가 난 적 있음 — 이모지를 텍스트에 넣었을 때 특히). 그냥 `https://...` 한 줄로 둘 것.
- 링크를 만들면 실제로 curl 등으로 200이 뜨는지 확인 후 전달할 것 (URL을 지어내지 말 것).

문서 전체를 그대로 슬랙 텍스트로 변환해야 하는 경우에만 아래 스크립트를 보조로 사용:
```bash
python scripts/md_to_slack.py <파일.md> [--out out.txt]
```

## Git / 배포 관련 주의사항

- V:는 NAS WebDAV 마운트. `git config core.fscache false` 필요(레포별 로컬 설정 이미 되어 있음)
- `Write`/`Edit` 도구가 가끔 "파일 크기가 다름" 경고를 내는데, 대부분 WebDAV 크기 확인 오차인 **false positive**. Read로 실제 내용을 확인하고 맞으면 그냥 진행할 것 (재작성 반복 불필요)
- **push 전에는 항상 사용자에게 확인받을 것** (public 레포에 반영되는 행위이므로) — 이건 예외 없이 매번 확인
- push 전에 `git fetch origin`으로 원격에 새 커밋(사용자가 GitHub 웹에서 직접 수정한 경우 등)이 있는지 확인 — 있으면 rebase 후 push
